# rsc conventions — wiring a skill into the catalog

A skill is not done when `SKILL.md` reads well. It is done when it is *installed* — in a bundle, in the marketplace, copied into the plugin tree, and indexed where the agent will find it. This reference is the rsc plumbing.

## The layout (single source of truth)

```text
skills/<id>/                    ← canonical source of truth; edit here, only here
  SKILL.md
  references/*.md
  evals/cases.yaml
  evals/README.md
  scripts/verify.sh             ← only if the skill has a checkable artifact

plugins/<bundle>/skills/<id>/   ← REAL COPIES, generated by sync-bundles.sh (never hand-edit)
plugins/<bundle>/.claude-plugin/plugin.json
.claude-plugin/marketplace.json ← the bundle catalog
scripts/sync-bundles.sh         ← copies skills/ → plugins/<bundle>/skills/
scripts/eval-lint.sh            ← gates evals/cases.yaml minimums
```

Rule: **`skills/<id>/` is the only place you edit.** `plugins/*/skills/*` are regenerated copies — editing them is a defect because the next sync overwrites your change.

## Bundles & namespacing

Skills are grouped into bundles; a skill is invoked as `/<bundle>:<id>`. Current bundles:

| Bundle | Holds |
| --- | --- |
| `rsc-core` | `init`, `harness` (+ `author-skill` — the meta tools that govern a workspace) |
| `rsc-backend` | `fastapi`, `go`, `postgresdb` |
| `rsc-frontend` | `nextjs`, `flutter`, `design` |
| `rsc-content` | `marketing`, `presentations`, `course-storytelling` |
| `rsc-agents` | `building-agents` |
| `rsc-ops` | `secure-coding`, `deployment` |

A new skill belongs in the bundle whose *job* it shares. The meta/governance skills (`init`, `harness`, `author-skill`) sit in `rsc-core` because they operate on the workspace and its tooling, not on a product stack. The SDD-phase skills ship as their own `rsc-sdd` bundle.

## Wiring steps for a new skill

1. **Add it to the bundle map** in `scripts/sync-bundles.sh` — the `skills_for_bundle` case for the target bundle gains the new `<id>`.
2. **Ensure the bundle's `plugin.json` exists** at `plugins/<bundle>/.claude-plugin/plugin.json` (version lives here only; `mcpServers: {}`; no agents/hooks fields unless the bundle truly needs them).
3. **Register the bundle in the marketplace** at `.claude-plugin/marketplace.json` if the bundle is new. An existing bundle needs no marketplace change — it picks up the skill on the next sync.
4. **Run `scripts/sync-bundles.sh`** so `plugins/<bundle>/skills/<id>/` becomes a real copy. Confirm **0 symlinks** — the plugins must be self-contained for tarball/zip/Windows distribution.
5. **Run `scripts/eval-lint.sh`** — it must report PASS for the new skill.
6. **Update the README catalog** (and the harness `claude-md-template.md` Knowledge-map rows) if the skill introduces a new topic users should discover.

```bash
# the two gates, from repo root
bash scripts/eval-lint.sh
bash scripts/sync-bundles.sh
# then confirm no symlinks slipped into the plugin copies:
find plugins -type l
```

## The Knowledge map

The root `CLAUDE.md` carries a `## Knowledge map` section that indexes the 02-DOCS wiki topics — it is what every other skill reads before working in its area (the `harness` convention). When a skill produces durable artifacts, they live under `02-DOCS/wiki/<topic>/` and get a Knowledge-map row. For SDD-related artifacts the topic is `02-DOCS/wiki/sdd/`. `author-skill` writes there only when a design note is worth keeping; the executable record is always the skill's own `evals/`.

## Real-copy bundles — why no symlinks

`sync-bundles.sh` does `rm -rf` then `cp -R` for each target, deliberately replacing any symlink with a real directory. Symlinked plugins break when zipped or unpacked on Windows, and break for users who install a single bundle without the rest of the repo. The cost is that you must re-sync after every edit; the benefit is a plugin that is genuinely standalone. Never "optimize" this back to symlinks.

## verify.sh — only for checkable artifacts

A `scripts/verify.sh` belongs in a skill that emits something a script can *check*: code (lint/type/test), config (schema), copy (a ban-list grep). It should be read-only by default and warn rather than fail unless asked to gate. **Process skills** — judged on the behavior/safety rails they install, like the SDD-phase skills and `author-skill` itself — have no artifact to grep, so they ship **no** `verify.sh`; their rigor is the `capability` eval. Adding a hollow `verify.sh` to a process skill is cargo-culting, not rigor.

## The originality rule (hard)

The rsc catalog is its own ecosystem, not a re-skin. When mining ideas from other skill libraries, take the *idea* and re-express it in the rsc voice. Never reproduce another ecosystem's signature artifacts or phrasing: no borrowed urgency blocks ("1% chance…"), no copied rationalization-table wording, no `*-reviewer-prompt.md` file convention, no verbatim flowchart text. Git authorship on rsc commits is Eric, never the assistant. If a draft reads like it came from somewhere else, rewrite it.
